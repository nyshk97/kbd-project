# 意思決定の記録

## プロジェクト目的：自分用の最高のキーボード

- このプロジェクトは自分用の最高のキーボードを作ることを目的にする
- 量産・販売・第三者向けキット化は目的にしない
- 一般的な使いやすさより、自分の手・作業環境・好みに合うことを優先する
- 部品単価、入手性、製造容易性は考慮するが、量産時の原価最適化やサプライチェーン安定性は優先しない
- 販売向けの認証、パッケージング、ユーザー向け組み立て性、販売後サポートのための設計は対象外にする
- BLE無線モジュールの使用地ごとの法令適合は、販売有無と別に部品選定時に確認する

## 進め方：段階試作

- 最終形を一度で作らず、簡単な試作を複数台作って段階的に近づける
- 各試作機は `boards/pXX-*` に分け、目的・部品表・作業ログを個別に持つ
- 最終形の設計データは `target/` に置き、試作で検証できた要素だけを反映する
- まだ使わないディレクトリは先に作らず、必要になった時点で追加する
- 試作をまたいで使うZMK設定、Ergogen定義、footprint、ケース部品が出てきたら `shared/` を作る

## ファームウェア：ZMK

- 完全無線・省電力・Auto Mouseレイヤー・エンコーダー対応が全部揃う
- QMKは有線前提のため非対応
- p01のZMK configはZMK CLIまたは選定したCorne Xiao系repoのサンプルconfigを起点にする
- current ZMKではXIAO nRF52840のboard targetは原則 `xiao_ble//zmk` として確認する
- 古い記事やrepoに残る `seeeduino_xiao_ble` などのboard名は、そのまま使える前提にしない
- Corne Xiao系のshield名は作者ごとに違うため、選定した基板repoの `boards/shields/` 定義を正とする
- 外部shieldはzmk-config内に置くか、ZMK moduleとして取り込む
- 分割構成ではcentral側がPCへHID出力する。peripheral側はPCと直接ペアリングしない
- p01のハードウェア正本は `JonMuller/gerbers` の `corne-choc-xiao` にする
- `LeducH/corne-choc-xiao` は同一Gerber/BOM/CPLの整理版として、ガイド・購入リンクの参考に使う
- p01のファームウェア出発点は `wintinue/zmk-corne` にする
- `wintinue/zmk-corne` の `chipper_left` / `chipper_right` shield定義を起点にし、`build.yaml` のboard名をcurrent ZMK向けに確認・修正する

## MCU：p01はXIAO nRF52840、targetはAuto-KDK専用無線ボード

- p01は無印XIAO nRF52840を使う（変更なし。既存のCorne Xiao系シールドをそのまま使える）
- targetはAuto-KDK採用（2026-07-15、下記PCB設計方針を参照）に伴い、専用無線コントローラボード（nRF52840、IO 18ピン・うち4ピンは拡張モジュールと共用）に変更する
- 専用ボードはBOOTH（のぎけす屋）で無線2個セット¥9,000。一体型・分割とも1セットでよい。コンスルー同梱ではんだ付け不要
- 電源はPHコネクタの1S Li-Po（保護回路必須）、充電200mA、バッテリー最大6×31×52mm
- 旧方針（target向けXIAO nRF52840 Plus候補）は不要になったため撤回

## 接続方式：まずBLE直結、必要になったらドングル追加

- 旧方針はBLEドングル前提（BLE直結はMacとの相性問題・スリープ復帰約5秒の懸念、ドングルはUSB常時接続でほぼ即時復帰）
- p02はBLE直結で使い始める方針に変更（2026-07-15）。ドングルの有無はハードウェアに影響せず、後からファームウェアのビルドターゲット追加＋手持ちの余りXIAO nRF52840の転用だけで移行できるため、スリープ復帰の遅さを実際に体感してから判断する
- ドングル構成がビルド可能かの検証はzmk-config-split-us60pctリポジトリのdongleブランチで実施（dongle/right_p shield、input-splitによるトラックボール中継、XIAOドングル対応）

## スイッチ：p01はChoc v1 Red、targetはChoc V2（19mmピッチ）

- 旧方針はChoc v1 Red（ロープロ最薄・Macの打鍵感に最も近い）だったが、Auto-KDKがChoc v1非対応のため覆した（2026-07-15）
- targetはChoc V2・19mmピッチ。MacBookのキーピッチ（約19mm）を維持して運指の記憶を移行する
- Choc V2はトータルトラベル約3.2mm（バリアントにより2.8mm）・アクチュエーション約1.3mmで、ロープロ最短クラス。MacBook（約1mm）の完全再現ではなく「メカニカルで到達できる最良の近似」と割り切る
- 軸（バリアント）はPCB・ケース設計に影響しないため後から決める。MacBook近似の観点で軽めリニア（Deep Sea Mini Pink Island 35gf / Islet 43gf / Red 45gf前後）を打ち比べ候補にする
- Choc V2はMXステム。キーキャップ自作はフラット・薄型・広めを狙うとMacBook感が出る
- Choc v1とV2は物理非互換（PCBセンター穴径が違う）ため、p01のPCBにV2は刺さらない。ホットスワップソケット自体は共通で、p01のソケット検品知見は活きる
- 2U以上のキーにはスタビライザーが必要。Choc V2対応ロープロスタビの入手性を確認し、入手困難なら2U以上のキーを置かないレイアウト（親指2Uを分割等）に逃がす

## トラックボールセンサー：PAW3222（target）

- 旧方針のPMW3610（＋SideraKBボード＋34mm固定）はAuto-KDKで選べないため、Auto-KDK標準のPAW3222に覆した（2026-07-15）
- PAW3222は低消費電力で、ZMKドライバ（zmk-driver-paw3222）を開発者sekigon-gonnoc氏自身がメンテしている。無線・バッテリー持ち重視というPMW3610選定理由をそのまま満たす
- ボール径34mm固定は撤回。Auto-KDKのバリエーション（thumb-25 / thumb-34 / index-25〜50）から実配置で選ぶ。径へのこだわりはない
- トラックボール使用時は無線ボードのIOを4ピン消費する（18→実質14）

## エンコーダー：搭載しない

- 旧方針はEVQWGD001（縦回転スクロールホイール）だったが、必要性が薄れたため搭載しない方針に変更（2026-07-15）
- Auto-KDKにエンコーダーの選択肢がない（EC11・EVQWGD001とも非対応）ことも整合する
- スクロールはトラックボール＋ZMKのスクロールレイヤーで代替する想定

## PCB・ケース設計方針：Auto-KDK

- 旧方針（Ergogen + FreeRouting + KiCad のフルスクラッチ）を覆し、targetはAuto-KDK（sekigon-gonnoc氏のWebツール https://auto-kdk.pages.dev/ ）で設計する（2026-07-15に調査のうえ決定）
- 決定理由: レイアウト指定からPCB（EasyEDA形式json→JLCPCB発注）・ケースSTL・ZMK shield一式（ZMK Studio対応）・フォームカットSVGまで自動生成でき、要件（分割・無線ZMK・ロースタッガード・トラックボール）をネイティブに満たす。特にケース設計の学習コストを丸ごと消せる
- 2026年6月ビルド（ver 2026-06-03）が稼働中のアクティブなプロジェクト。製造到達したユーザー事例も複数あり
- 制約として受け入れたもの:
  - 専用コントローラボード必須（XIAO等の市販MCUボードは選べない）
  - 無線はrow+col合計18ピン上限。トラックボール使用側は実質14ピンで、5行×9列だと上限ちょうど（キー追加不可）
  - PCBデータはEasyEDA形式のためKiCadでの手直し不可。手直しはEasyEDA上で行う
- 未解決事項の確認結果（2026-07-15、データ生成zipで確認）:
  - ライセンス: 解決。利用規約により個人利用・商用利用とも可。ただし生成データは**Auto-KDK公式コントローラとの組み合わせでのみ使用可**。現状有姿・無保証
  - ホットスワップ: 解決。生成PCBにKailh Chocソケット（CPG135001S30）×64 + 1N4148W×64のフットプリントあり。はんだ付けは自分で行う
  - 左手PCB: 解決。left/right/panelized（面付け版）の3種のEasyEDA jsonが生成される
  - スタビライザー: フットプリントなしと判明。Choc V2構成では2U/2.25Uキー（左Shift 2.25U・左親指2U・右2U・右Enter 2.25U）がスタビなしになる → **スタビなしのまま行くと決定（2026-07-15）**。ロープロ＋トラベル3mm＋軽量キャップなら実害は小さい見込みで、「スタビなし幅広キーが実用になるか」自体をp02の検証項目にする。ダメなら次イテレーションで幅広キーを1.5U以下に分割（既存マトリクスの空きセルに収まるためピン数制約には抵触しない）
  - ドングル対応: 解決。zmk-config-split-us60pctのdongleブランチで、dongle shield（central役・モックkscan）＋right_p shield（右手peripheral化・トラックボールをzmk,input-splitで中継）を追加し、akdk_bt1ドングル/XIAOドングル（ZMK v0.3では board名 `seeeduino_xiao_ble`。`xiao_ble//zmk` は新しいZMK mainの名前でv0.3では無効）とも全ターゲットビルド成功（2026-07-15）。必要になったらこのブランチをmainに取り込むだけ
  - 自動配線の品質: 未検証（EasyEDAインポート時に確認）
- 旧方針のErgogen成果物（キー配置・1:1印刷アウトライン）とKLE基準は、レイアウト検討の参考資料として残す

## キーマップ：トラックパッドジェスチャの代替はZMKコンボ

- トラックパッド廃止に伴い、4本指スワイプ相当の操作はmacOS標準ショートカット（Ctrl+矢印）をコンボで送る（2026-07-15）
  - J+K = Ctrl+←（左のスペースへ）
  - K+L = Ctrl+→（右のスペースへ）
  - J+K+L = Ctrl+↑（Mission Control。トグルなので閉じるのも同じコンボ）
- Ctrl+↓（App Exposé）は不要と判断。恋しくなったらM+,+.（JKLの真下段）に追加する
- 誤発動対策はrequire-prior-idle、3キー同時が押しにくければtimeout-msを50→80ms程度に調整
- コンボはZMK Studioでは編集できないため`.keymap`に記述してGitHub Actionsでビルドする。レイヤー・キー割り当ての試行錯誤はStudio、確定したコンボはkeymap、という分担

## ケース（フェーズ1）：3Dプリント

- 動作確認フェーズは速度・コスト優先
- DMM.make or JLCPCBに発注
- フェーズ2の素材（アルミ削り出し等）はフェーズ1完了後に検討
- マウントタイプはインテグレーテッド（2026-07-15）。追加部品不要で試作が速く、硬めの打鍵感はMacBook近似の目的とも整合する。ガスケット/Tadpoleの柔らかさが欲しいかはp02を打ってから判断
- Choc V2用の土手（switchEmbankment）はオフで確定（2026-07-15）。見た目と防塵だけの違いで打鍵感に影響しないため、気が変わったらケース再印刷で対応
- p02ケースパラメータ（2026-07-15確定）: チルト5°、テント4°、底面厚さ3mm（強度優先で1.8mmから戻した）、底面縁幅5mm、上面縁幅3mm、角の種類=丸型、R面取り（上部2mm、下部3mm=接地際を削って薄く見せる）。MacBook寄りの柔らかい外観狙い。全てケースSTLのみに影響するため、実物を見て気に入らなければ再印刷で調整

## キーキャップ：YUZU Custom Keycaps（PFFプロファイル）を本命に

- 旧方針（Choc v1の印字ありキャップは市販品が少ない→フェーズ1は印字なしMBK＋3Dプリント自作で割り切り、印字は後で検討）は、Choc V2化＋YUZUで課題ごと解消したため覆した（2026-07-15）
- YUZU Custom Keycaps（https://yuzukeycaps.com/ 、製造はKeyreative）で、自分のレイアウト・印字あり・1.5mm厚PBT・昇華印刷のキャップを少量注文できる。1キー$0.75〜（64キーで$48〜＋送料）、納期20〜30日
- プロファイルは**PFF**（matt3o設計、高さ5mm、Choc V2/Gateron LP専用のロープロ、全段同形）を本命にする。フラットでMacBook近似の目的に合い、Cherry彫刻より背が低い。PFFはChoc v1非対応
- Auto-KDKの「キーキャップ(YUZU)」連携でレイアウトを持ち込める。キーごとのRowIndex（R1〜R4=Cherry段指定、R0=KAM）はCherry/KAM用で、PFFは全段同形のため実質不要（PFF注文時の連携挙動はYUZU画面で要確認）
- 発送前の製造確認写真は必ずチェックする（印字位置ミスが稀にあるとのレビューあり）
- Auto-KDK生成のyuzu-set.jsonをYUZUに読み込み、プロファイルをPFFに切り替えて確認済み（2026-07-15）: 必要な全キー幅（1.25u/1.5u/1.75u/2u/2.25u）とhomingキーがPFFで揃う。64キーで$63.90（KAMと同額）。注文はPCB等の納期に合わせて後日でよい
- 3Dプリント自作（YUZUのSTLではなく自前モデリング）は、造形自由度が必要になった場合の予備手段に格下げ
